sidekiq-unique-jobs v9: A Complete Rewrite

After months of incremental improvements, bug fixes, and listening to production issues from hundreds of users, I decided it was time to stop patching and start over. sidekiq-unique-jobs v9 is a ground-up rewrite of the locking core — fewer Redis keys, fewer lines of code, and meaningfully faster.

Why a rewrite?

The v8 architecture accumulated complexity over years of feature additions. Every lock created 13 Redis keys — a STRING for the digest, LIST keys for QUEUED and PRIMED states, a HASH for locked JIDs, a STRING for lock INFO metadata, a full set of :RUN duplicates for runtime locks, plus entries in three sorted sets (digests, expiring_digests, and changelog). The QUEUED/PRIMED keys existed solely for a BLMOVE-based blocking lock mechanism that most users never needed.

This complexity was the root cause of the most persistent issues: locks not releasing after shutdown, orphaned digests growing forever, the reaper fighting race conditions across 4 different Redis data structures, and silent job failures when lock TTLs expired.

I looked at how Sidekiq Pro's SuperFetch works, studied sidekiq-reliable-fetch and GitLab's fetcher, and realized the right approach was to build the lock management into the fetch layer itself — not as middleware bolted on after the fact.

What changed

2 Redis keys instead of 13

v9 stores locks using just two Redis keys:

  • digest:LOCKED — a Hash mapping job IDs to JSON metadata (worker, queue, lock type, timestamp)
  • uniquejobs:digests — a Sorted Set tracking all active lock digests

Everything else is gone. No QUEUED lists, no PRIMED lists, no INFO strings, no :RUN duplicates, no separate expiring_digests set, no changelog. The LOCKED hash is the source of truth.

Simpler Lua scripts

v8 had 15 Lua scripts with debug logging, shared partials, and changelog writes on every operation. v9 has 8 scripts, each under 40 lines, with zero debug overhead:

-- lock.lua: the entire lock acquisition script
if redis.call("HEXISTS", locked, job_id) == 1 then
  return job_id  -- idempotent re-lock
end

if redis.call("HLEN", locked) >= limit then
  return nil  -- contention
end

redis.call("HSET", locked, job_id, metadata)
redis.call("ZADD", digests, score, digest)

if pttl and pttl > 0 then
  redis.call("PEXPIRE", locked, pttl)
end

return job_id

That's it. No QUEUED/PRIMED dance, no BLMOVE, no sentinel pushes.

ReliableFetch: atomic job tracking

v9 includes an optional ReliableFetch strategy that uses Redis LMOVE to atomically move jobs from the queue to a per-process working list. This means:

  • No job loss on crash — jobs are tracked in the working list until acknowledged
  • Lock validation at fetch time — the Lua fetch script checks if the lock is still valid in the same atomic operation
  • Crash recovery on startup — scans for dead process working lists and requeues their jobs
  • Lock-preserving shutdown — during graceful shutdown, locks persist for requeued jobs
# Configuration
Sidekiq.configure_server do |config|
  config[:fetch_class] = SidekiqUniqueJobs::Fetch::Reliable
end

The reaper became trivial

v8's reaper was 615 lines across 6 files — a Manager, Observer, ReaperResurrector, base Reaper class, RubyReaper, and LuaReaper. It had to scan queues, retry sets, schedule sets, and process sets for every digest to determine if a lock was orphaned.

v9's reaper is ~30 lines inlined in the Server class. It calls a 27-line Lua script that does one thing: scan the digests sorted set, check if each digest's LOCKED hash still exists, and remove it if not. That's it — because the LOCKED hash is the source of truth, and Redis TTL handles expiration.

Lock metrics

v9 ships with built-in metrics tracking per lock type. The Locks tab in the Sidekiq Web UI now shows a summary table:

Lock Type Acquired Denied Released Failures
until_and_while_executing 847 12 835 0
until_executed 2,341 3 2,338 0
total 3,344 15 3,329 0

Metrics use the same pattern as Sidekiq's built-in ExecutionTracker: thread-safe in-memory counters flushed to Redis every 60 seconds, with 8-hour retention and auto-expire.

Performance

Benchmarked on the same machine, same Redis, same Ruby 3.4.7 + YJIT + Sidekiq 8.1.1:

Benchmark v8.0.12 v8.1.0 v9.0.0 Improvement
Lock + Unlock 6,520 i/s 6,855 i/s 8,684 i/s +33%
Lock + Execute 8,445 i/s — 14,406 i/s +71%
No contention 1,805 i/s ~6,855 i/s 8,019 i/s +344%
High contention 2,920 i/s ~5,400 i/s 5,461 i/s +87%
Memory (100 execute) ~2.8M 2.45M 1.33M -52%
Redis keys per lock 5+ 3+ 2 -60%
Keys after unlock multiple multiple 0 Clean

The biggest wins are in the execute path (+71% throughput, -52% memory) which is the hot path for every Sidekiq job. The no-contention improvement (+344% from v8.0.12) comes from eliminating the QUEUED/PRIMED/BLMOVE round-trip that every lock acquisition previously required.

Breaking changes

  • Sidekiq 8+ required (dropped Sidekiq 7)
  • Ruby 3.2+ required (dropped 2.7, 3.0, 3.1)
  • Changelog removed — use the reflection system for lock event observability
  • Expiring locks unified — no separate expiring_digests sorted set; TTL-based locks use the same digests ZSET with expiry time as score
  • Web UI simplified — single "Locks" tab with metrics (removed Expiring Locks and Changelogs tabs)

Upgrading from v8

Just bump the gem version. v9 automatically migrates v8 lock data on first startup — it scans for old keys (QUEUED, PRIMED, INFO, :RUN variants), removes them, merges expiring_digests into digests, and preserves all active LOCKED hashes. No manual steps, no downtime.

# Gemfile
gem "sidekiq-unique-jobs", "~> 9.0"

Configuration is unchanged — sidekiq_options lock: :until_executed works exactly as before.
